Testing Static Site Generators for a Personal Knowledge Base

2026-07-12

static-site, knowledge-base, experiments

<h1>Testing Static Site Generators</h1> <p>I tested a few static site generators for my AI knowledge base. Hugo was fast but added complexity I did not need. Jekyll required a Ruby dependency I did not want to install. I chose a simpler approach using Python and Jinja2 because reducing dependencies made the system easier to understand and maintain.</p> <h2>What I tried</h2> <ul> <li><strong>Hugo</strong>: fast, but the template language was overkill for a small site.</li> <li><strong>Jekyll</strong>: solid, but I did not want to install Ruby just to build a site.</li> <li><strong>A custom Python script</strong>: about 200 lines, easy to read, does exactly what I need.</li> </ul> <h2>What I learned</h2> <p>The simplest tool that solves the problem is usually the best choice for a small project. Fewer dependencies mean fewer things to break, and the whole system stays auditable — you can read every line of the build script and understand what it does.</p> <h2>Why this matters</h2> <p>A knowledge base is a long-term project. If the build tool is something you don't understand, fixing it when it breaks becomes a project of its own. A 200-line Python script using libraries that have been stable for years is something I can maintain myself, without waiting for a framework update or debugging a template language I barely use.</p>
